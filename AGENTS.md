# 3 Leaps Homebrew Tap - AI Agent Guide

## Operating Model

This repository is intentionally small. Treat it as distribution infrastructure, not a product codebase.

- Default role: `devlead`
- Escalate to maintainers for release timing, public publishing decisions, and any change that affects multiple upstream repositories
- Keep changes narrow: formulae, tap automation, and maintainer docs only

## Role Catalog

This repo does not carry a full `config/agentic/roles/` tree.

Use the shared catalog in `~/dev/3leaps/crucible/config/agentic/roles/` when a role definition is needed. In practice:

- `devlead` for formula and automation work
- `releng` for release-process changes
- `devrev` for code review
- `secrev` for signing, provenance, or key-handling changes
- `deliverylead` only for cross-repo delivery planning, not implementation

## Repository Purpose

This tap publishes Homebrew formulae for CLI tools released from the `3leaps` GitHub organization.

Current responsibilities:

- maintain formulae in `Formula/`
- keep maintainer docs current
- provide local automation for formula updates
- track real published GitHub release assets

Out of scope:

- upstream build pipelines
- release signing implementation inside app repos
- packaging for Windows
- broad organizational planning docs

## Files That Matter

- `Formula/*.rb` - Homebrew formulae
- `scripts/update-formula.sh` - release-to-formula updater
- `Makefile` - local maintainer workflow
- `README.md` - user-facing tap entry point
- `RELEASE_PROCESS.md` - maintainer release flow
- `docs/adding-a-formula.md` - onboarding pattern for future tools

## Workflow

### Before Editing

1. Read `README.md` and `RELEASE_PROCESS.md`
2. Confirm the target app has a published GitHub release with macOS/Linux assets
3. Prefer updating the generator script over hand-editing multiple formulas when the pattern changes

### Before Committing

1. Run `make style`
2. Run `make audit APP=<app>` for touched formulae
3. Run `make test APP=<app>` for touched formulae
4. Keep docs aligned with the actual release pattern

## Platform Policy

Assume the current 3 Leaps Homebrew baseline is:

- macOS ARM64
- Linux AMD64
- Linux ARM64

Intel macOS is optional per tool and should only appear when the upstream repo actually publishes that asset.

Windows assets may exist upstream, but they are not consumed by Homebrew.

## Commit Attribution

Use the standard 3 Leaps attribution format:

```text
<type>(<scope>): <subject>

<body>

Generated by <Model> via <Interface> under supervision of @3leapsdave

Co-Authored-By: <Model> <noreply@3leaps.net>
Role: <role>
Committer-of-Record: Dave Thompson <dave.thompson@3leaps.net> [@3leapsdave]
```

When creating commits from the shell, use separate `-m` paragraphs or an editor-backed message. Do not embed literal `\n` escape sequences in the commit body.

## Guidelines

### Do

- keep the repo lean
- use live GitHub release metadata when updating formulas
- document cross-repo release steps clearly
- preserve Homebrew tap naming conventions (`3leaps/tap`)

### Do Not

- add a heavy local role catalog unless repo scope materially expands
- assume every tool supports every platform
- change upstream release behavior from this repo
- commit secrets, keys, or local environment details
